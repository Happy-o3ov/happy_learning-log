# AI 모의면접 프로젝트 – 결과 페이지 구조 정리 기록

> 날짜: 2026-02
>
> 프로젝트: AI Interview Simulation (Frontend 중심)

---

## 1. 작업 배경

AI 모의면접 프로젝트를 진행하면서 **결과 화면 구조가 직관적으로 설명되지 않는 상태**에 놓여 있었다.

결과는 정상적으로 출력되고 있었지만,
- 결과 페이지가 여러 개로 보였고
- 관리자/면접자 화면의 경계가 불분명했으며
- 라우팅 기준이 명확하지 않아 스스로도 구조를 설명하기 어려웠다.

이번 작업의 목적은 기능 추가가 아니라,
**이미 구현된 결과 화면들을 ‘역할 기준’으로 재정리하는 것**이었다.

---

## 2. 기존 결과 화면 구성

프로젝트에는 결과 화면이 총 3가지 형태로 존재했다.

### 2.1 면접 종료 후 결과 모달

- 위치: `/interview`
- 형태: React 컴포넌트 기반 Modal
- 역할:
  - 면접 직후 즉각적인 피드백 제공
  - 사용자의 흐름을 끊지 않기 위한 UX 목적

### 2.2 면접자용 단독 결과 페이지

- 경로: `/result/:threadId`
- 형태: React Router 기반 페이지
- 역할:
  - 면접 종료 후 결과 재확인
  - PDF 저장 기능 제공
  - 외부 링크 공유 가능

### 2.3 관리자용 결과 페이지 (정적 페이지)

- 형태: `result.html + result.js`
- 기술 스택:
  - Vanilla JS
  - Chart.js (Radar Chart)
- 역할:
  - 관리자 관점의 평가 분석
  - 근거(rationale) 노출
  - 시각화 중심 리포트

---

## 3. 혼란의 원인 분석

혼란의 핵심 원인은 **관리자 결과 페이지의 성격 오해**였다.

관리자 결과 페이지는:
- React 컴포넌트 ❌
- SPA 내부 라우트 ❌
- React Router 대상 ❌

즉, 프론트엔드 앱의 일부가 아니라
**독립 실행형 리포트 페이지**였다.

하지만 이를 React Router 경로(`/admin/results/:id`)로
통합하려고 시도하면서 다음 문제가 발생했다.

- 화면이 렌더링되지 않음
- 데이터 로딩 로직 불일치
- 역할별 페이지 구분 실패

---

## 4. 해결 전략

### 4.1 핵심 원칙

> **결과 데이터는 하나, 결과 표현(View)은 여러 개일 수 있다.**

이에 따라 결과 페이지를 억지로 통합하지 않고,
각 페이지의 책임을 명확히 분리했다.

### 4.2 적용한 구조

- 면접자 결과
  - React Router 기반 유지
  - Modal / 단독 페이지 병행

- 관리자 결과
  - React Router에서 제거
  - Admin Dashboard에서 `window.open()`으로 접근
  - 기존 Chart.js 기반 구조 유지

이 방식으로:
- 기존 코드 대부분을 유지하면서
- 역할 기준 구조를 명확히 할 수 있었다.

---

## 5. 라우팅 설계 관점에서의 정리

이번 작업을 통해 다음을 명확히 인식했다.

- 라우트는 단순한 URL 문자열이 아니다
- 라우트 하나가 사용자 흐름을 결정한다
- 버튼에서 직접 판단하지 않고, 라우트에 역할을 위임해야 한다

### 적용 기준

- 로그인 시 `role`만 설정
- 화면 이동은 역할별 진입 경로로 제한
- 관리자/면접자 결과 페이지를 물리적으로 분리

---

## 6. 결과 및 회고

이번 작업은 새로운 기능을 추가한 작업은 아니지만,
프로젝트 구조를 **설명 가능한 상태**로 만드는 중요한 단계였다.

### 얻은 교훈

- 화면이 많다고 해서 잘못된 설계는 아니다
- 헷갈린다는 것은 구조 정의가 부족하다는 신호다
- 기능 구현보다 **역할 정의와 책임 분리**가 더 중요할 수 있다
